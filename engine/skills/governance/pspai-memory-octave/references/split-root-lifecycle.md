# 裂根问题完整生命周期

> 来源：L6悟道#002→#003→#004 跨轮追踪（2026-06-01→2026-06-03）
> 性质：MEMORY.md文件与memory工具脱钩的演化全貌
> 关联：`⚠️ 裂根警告` 章节（pspai-memory-octave SKILL.md）

---

## 什么是裂根

MEMORY.md（磁盘文件）和 memory 工具（Hermes 内置 KV 存储）是两套独立系统，不存在自动同步机制。这种"一个逻辑记忆层有两个物理存储"的设计就是裂根。

| 系统 | 管理方式 | 配额上限 | 可被 L7 cron 修 | 可被对话修 |
|:--|:--|:--|:--|:--|
| MEMORY.md | `write_file` / `patch` | 无硬限 | ✅ | ✅ |
| memory 工具 | `memory.add/replace/remove` | 8,000 chars | ❌ | ✅ |

## 演化三阶段

### 阶段一：容量失控（L6#002，2026-06-01）

```
MEMORY.md: 16,208B (2.64×硬限)
memory工具: 39% (~3,120 chars)
→ 文件暴涨但工具配额低 → 迁移阈值(80%)永不触发
→ L1→L3管道对文件层面的堆积无感知
→ 裂根表现为「容量失控」
```

**触发条件**：大量对话内容直接写入 MEMORY.md（里程碑、产品决策）绕过 memory 工具配额。

### 阶段二：双轨愈合（L6#003，2026-06-02）

```
MEMORY.md: 12,001B (L7清理后)
memory工具: ~10%
→ L7 cron可处理文件层问题（归档+短指针替换）
→ 裂根表现为「文件-工具双轨，L7部分愈合中」
```

**暴露的新问题**：L7 清理速度（~4KB/次）vs 日常新增速度（~1-3KB/天）→ 稳态可维持。但突发涌入（单日+8KB）超过单次清理能力。

### 阶段三：知识丢失（L6#004，2026-06-03）

```
MEMORY.md: 1,127B (对话清理后，-91%)
memory工具: ~10%
→ 对话清理走write_file全量替换 → 不经过归档检查
→ 6+知识块（产品战略/架构/UI规范/凭证）未进L3即丢失
→ 裂根表现为「内容被删而工具无感知 → 知识净流失」
```

**核心发现**：裂根的本质不是「两个存储不同步」，而是 **MEMORY.md 是「无保护的写入目标」**——任何 agent 都可以全量替换它而不触发任何归档检查。

## 裂根的终极公式

```
裂根 ≠ 两个存储不一致
裂根 = write_file(MEMORY.md) 可以跳过所有归档机制
```

**正确链路**：内容变更 → 检查是否应归档 → 归档 → 替换指针 → 完成
**裂根链路**：内容变更 → 完成

## 为什么三个阶段顺序不可逆

1. 阶段一必须先于阶段二：文件必须膨胀到 > 6KB 才触发修复需求
2. 阶段二必须先于阶段三：L7 的清理能力被确认（~4KB/天）后才暴露"对话清理更快且更破坏"
3. 阶段三暴露终极缺陷：即使 L7 完美运行，对话清理仍可绕过它

## 修复优先级

| 优先级 | 修复 | 解决阶段 |
|:------:|:-----|:------:|
| 🔴 P0 | MEMORY.md 写前归档检查 | 阶段三 |
| 🟡 P1 | l1_memory_guard.py check_file 模式 | 阶段一 |
| 🟡 P1 | 双指标迁移触发（文件≥10KB OR 工具≥80%） | 阶段一 |
| 🟡 P1 | 对话清理改为迁移标记 | 阶段三 |
| 🟢 P2 | 单一真相源（二选一） | 全部 |

## 教训

1. **双重写入目标 = 注定不一致**。两个独立的存储用于同一逻辑目的时，不一致是必然的，同步是偶然的
2. **监控盲区 = 攻击面**。l1_memory_guard.py 只监控 memory 工具 → 文件层面的问题完全不可见
3. **修复不能只修一个阶段**。修了阶段一（L7 可清理文件），暴露出阶段三（对话清理更危险）。裂根需要系统性修复而非逐阶段修补
