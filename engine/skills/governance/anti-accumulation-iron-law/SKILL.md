# 🛡️ 防积压铁律（昱纪通〔2026〕第02号 · 兄弟Agent起草）

## 背景

AI 智能体的上下文窗口是有限的。对话历史、记忆注入、技能加载全部挤在 LLM 的 token 预算里。不加控制，24 小时内必然积压→溢出→认知衰退——越聊越傻、越错越多。

## 三条永不妥协

### 一、不记录垃圾
写入永久记忆前必须通过三道筛子：
- **类型筛**：只收 skill/fact/insight/reflection/rule/learn
- **长度筛**：内容 ≥50 字
- **去重筛**：已有相同内容不重复写入

dialogue/cycle/audit/heartbeat 一律不入库。

### 二、不扫描全量
只在用户消息含以下**知识触发词**时才检索记忆：
`查/搜/分析/总结/怎么/为什么/对比/评估/回顾`

问候（"在吗"/"你好"）、确认（"好的"/"继续"）、命令（"写文件"/"重启"）一律不触发记忆检索。

### 三、不自动注入
LLM 窗口每次只带：
- 最近 **30 轮**对话
- 最多 **3 条**相关知识

旧对话不混入新上下文。记忆只按需检索，不全量注入。

## 上下文压缩机制（对标 Hermes Agent context_compressor.py · 默认50%）

Hermes Agent 的 context_compressor.py 默认阈值是 **0.50（50%）**，不是文档示例中的85%。算法五步：裁剪旧工具结果→保护头消息→保护尾消息(~20K tokens)→结构化摘要中间轮次→后续迭代更新。

PSPAI对标：上下文≥50%时先压缩🗜️，压缩后仍超限再截断。压缩原则：保留操作结果，压缩过程描述。

| 参数 | 值 | 说明 |
|:-----|:--:|:-----|
| compress threshold | 50% | 对标 context_compressor.py 默认值 |
| protect_first_n | 3 | 保护前3条消息 |
| protect_last_n | 20 | 保护后20条消息 |
| summary_ratio | 20% | 摘要占压缩内容的20% |

### AGENTS.md 自动注入毒药（2026-05-29 发现）

Hermes Agent 自动扫描工作目录下的 `AGENTS.md`/`CLAUDE.md`/`.cursorrules` 并注入每次对话。`hermes-agent/AGENTS.md`(20KB) 每次都被注入，是上下文最大杀手。修复手段：
1. `~/.hermes-yulong/AGENTS.md` 空文件覆盖
2. config.yaml 加 `skip_context_files: true`

| 参数 | 值 | 说明 |

| 持久化上限 | 200 条 | 硬盘存储上限 |

## 四级预警截断（对标 Hermes Agent 上下文颜色码）

| 级别 | 上下文 | 时间 | 轮次 | 动作 |
|:---|:---|:---|:---|:---|
| 🟢 绿色 | <50% | <45分钟 | <15轮 | 正常 |
| 🟡 黄色 | 50-80% | >45分钟 | >15轮 | 内部标记，加倍验证；≥50%触发压缩 |
| 🟠 橙色 | 80-95% | >90分钟 | >30轮 | 先压缩🗜️，压缩后仍超限再截断 |
| 🔴 红色 | ≥95% | >120分钟 | >40轮 | 立即截断，不许犹豫 |

情绪触发（直接红色）：用户说"你傻了/语无伦次/思维混乱"、连续3次操作失败、被否定≥2次。

每10轮自检四问：第几轮？开了多久？操作成功了吗？用户情绪正常吗？

核心铁律：**宁可多截断，不可硬撑。** 截断只断一个任务，溢出毁掉所有能力。

## L1→L3 迁移过滤（2026-05-29 新增）

1. 已被L4覆盖 → 不重复迁移，压缩为指针指向L4经验
2. 纯状态记录 → 不入L3，丢弃
3. 已被证明错误 → 加删除线 + 新信息链接后再迁移
4. 同主题碎片化条目 → 合并后再迁移，不超过3条同主题L3文件

## ⚠️ 常见陷阱（2026-05-29 实战教训）

### SOUL.md ≠ config.yaml system_prompt
- SOUL.md 和 config.yaml 的 system_prompt 是**两套独立文本**，不存在自动同步
- 改完 SOUL.md 后必须同步更新 config.yaml 的 system_prompt
- 更新后必须**重启网关**（systemctl --user restart hermes-yulong.service）

### 跨会话任务残留
- 新会话是白纸，不自动承接上一轮的未完成任务
- 除非用户明确说"继续上次的XX"，否则不自动执行旧任务
- 这是上下文溢出的常见伪装——把旧对话任务当成当前指令

### 双网关冲突
- 同机可能有多个Hermes网关实例（如兄弟Agent的独立通道）
- 由 systemd 管理，kill 后会自动重启
- 两个网关监听同一飞书频道可能抢消息

### 轮次自检未执行
- 铁律规定了每10轮自检四问，但"靠心里记住"完全不可靠
- **硬方案**：MEMORY.md 维护 `turn_count`，每次回复后+1
- SOUL.md 写入触发点：第10轮提醒、第20轮黄色、第28轮主动预警用户、第30轮截断
- **教训**：软规则不执行，必须代码级硬连

## 四道防线（代码级，不可绕行）

- **写入门**：record()/save() 前必须过类型+长度+去重三道筛子
- **检索门**：think 前先判消息类型，非知识类跳过全部记忆检索
- **窗口门**：对话历史 ≤30 轮，压缩阈值80%，持久化 ≤200 条，检索最多 3 条
- **自检门**：每 10 轮输出计数。每日凌晨深度清理+GC

## 自问三句

1. 历史超 40 了吗？
2. L3 文件数正常吗？
3. 缓存清了吗？
