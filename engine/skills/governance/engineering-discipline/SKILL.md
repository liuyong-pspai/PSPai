# 工程纪律

## 接到任务后的步骤

**第0步：多阶段计划必须文件持久化**

当用户给出含多个阶段/环节的完整计划时，**立即写入文件**——不靠脑子记、不靠上下文撑。
```
# 写入工作计划
write_file → plans/当前任务.md
内容：阶段清单 + 每个阶段的目标 + 完成标志
```

**为什么：** 上下文截断是不可预测的。30轮对话后旧计划自动消失。没有文件 = 丢失计划 = 用户问进度时答不上来。这不是技术问题，是工程纪律问题。

**自检：** 每完成一个环节，更新计划文件的完成状态。文件在 = 计划在。

**第1步：确认意图**
- 用户说的话，真的听懂了吗？
- 如果不确定 → 快速确认后再动手
- 禁止："先试试看"的试探式编码

**第2步：改前备份**
```bash
cp file.py file.py.BACKUP_$(date +%Y%m%d)
```
- 只改要改的文件，不改的别碰

**第3步：改完验证**
- 语法检查：`python3 -m py_compile file.py`
- 逻辑验证：测试正向路径 + 异常路径 + 边界值
- Web UI 部署后：加载静态资产完整性检查技能，运行资产完整性脚本——从HTML提取所有引用路径，逐条HTTP可达性验证

**第4步：如果验证不通过**
- 立即回滚 → 重新分析根因 → 再改
- 不修修补补凑合过去

**第5步：汇报**
- 汇报格式：结论 → 原因 → 方案 → 关键数据
- 简明扼要，不写长段落

## 大型文件拆分（Mixin 提取）铁律

从巨型文件提取 Mixin 子模块时的标准流程：

**前置：备份**
```bash
cp engine_tools.py engine_tools.py.BAK
```

**步骤：**
1. 确定提取范围（精确到行号：起始行→结束行）
2. 创建独立子模块文件（类名 + 属性）
3. **不要用 `patch` 做大段删除**——patch 只能匹配精确文本，大段代码无法可靠匹配
4. 用 Python 脚本做转换：读取全部行 → 切片重组 → `write_file` 写回
5. 双向语法验证：`python3 -m py_compile` 新模块 + 修改后的原文件
6. 更新计划文件记录进度

**反模式1：grep行号差 ≠ 代码行数**

用 `grep -n` 两个方法之间的行号差估算代码量 → 系统性高估 5-10 倍。

正确做法：做拆分计划前，必须 `read_file(offset, limit)` 实际读取目标区域，确认真实边界后再定计划。

**反模式2：patch 删除大段代码**

patch 只能匹配精确文本，超过 50 行的代码块无法可靠匹配。用 Python 脚本做切片重组。

**反模式3：Mixin 导入断裂**

从类中提取方法到 Mixin 后，Python 在方法定义所在模块的全局作用域解析名称。拆分后 Mixin 文件即使继承链正确，运行时仍可能 NameError。

| 陷阱 | 示例 | 后果 |
|:---|:---|:---|
| Mixin 文件零 import | 方法用 `log.info()` 但文件无 import | 调用即崩溃 |
| 裸名类变量引用 | 直接引用父类变量而非 self.xxx | NameError |
| 父类硬编码引用 | 硬编码父类名 | 改用 type(self) |

修复清单（每个新 Mixin 文件必须过）：
- 每个外部调用必须对应一个顶层 import
- 父类硬编码全部替换为 type(self)
- 两个文件之间无循环导入
- python3 -m py_compile 通过 ≠ 安全，必须独立验证每个 Mixin 文件的 globals 依赖

**反模式4：审计误判——把"默认启用"判定为"默认关闭"**

安全审计必须在审计报告交付前，对每个发现逐行回到源代码验证。条件表达式（`==`、`!=`、`or`、`and`）是误判高发区。

**反模式5：execute_code 中用 read_file → write_file 编辑文件**

在 `execute_code` 中使用 read_file 读取文件内容后直接写回 → 文件被嵌入行号，静默损坏。

| 操作 | 是否安全 | 原因 |
|:---|:--:|:---|
| execute_code + read_file → write_file | ❌ | read_file 输出含行号前缀 |
| patch 工具 | ✅ | 原生工具，不做行号编码 |
| execute_code + 全新内容 → write_file | ✅ | 不涉及 read_file |

文本编辑只用 `patch` 工具。execute_code 里可以读文件分析，但绝不把 read_file 的输出写回文件。

**反模式6：只看表面功能，不审深层实现**

采纳外部代码或评审工具时，只看表面功能不读源码 → 引入隐藏污染。

| 表面看到 | 深层实际 | 影响 |
|:---|:---|:---|
| "多引擎搜索" | 硬编码关键词追加 | 所有搜索被污染 |
| "支持中文" | 依赖脆弱的 class 名匹配 | DOM改版即失效 |
| "有标签功能" | 函数定义了但从未调用 | 功能死代码 |

正确做法（三问审计）：
1. 通读源码：每条关键逻辑路径都要看，不看描述看代码
2. 实测验证：用真实查询跑一遍，对比预期和实际
3. 找隐藏假设：硬编码字符串、死代码、未使用的函数

## 排查问题的方法

```
问题出现
  ↓
① 确认现象（是什么、什么时候开始的、影响范围）
② 查日志（不靠猜）
③ 最小化问题（只改最少的代码来验证根因）
④ 写下假设 → 验证 → 证伪或证实
⑤ 修复（一步到位，不试探）
⑥ 回归验证（确认修复后问题不复发）
```

## 一次只改一个问题

- 不要在同一个改动里同时修多个问题
- 分批改，分批验证
- 出问题了容易定位，回滚成本低

## 牢记

1. 每个改动都要对用户负责
2. 改坏了不可怕，不改回来才可怕
3. 用户的时间比试错成本值钱 — 想清楚再动手
