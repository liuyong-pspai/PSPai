# 自我审计方法论

## 第0步：会话健康检查

在审计代码之前，先审自己：
- **会话时长**：超过8小时 → LLM上下文已超载，输出质量断崖下跌
- **历史轮数**：超过80轮 → 已触发防积压铁律上限
- **自问三句**：历史超100了吗？L3文件数正常吗？缓存清了吗？

发现超标 → 立即建议用户新建对话。不在超载上下文里硬撑。

## 第1步：代码级审计
- **语法**：`python3 -m py_compile file.py` 通过
- **安全**：没有shell注入、没有exec无沙箱、没有open缺encoding
- **异常处理**：没有bare except + pass吞异常
- **并发**：共享变量有锁保护

## 第2步：架构级审计
- **紧凑化闭环**：输入→处理→输出→反馈链路完整
- **对话历史重建**：重启后上下文可恢复
- **按需技能加载**：工具/技能有没有正确注册
- **结构化验证**：有没有语法检查、参数校验

## 第3步：客户视角
站在用户的角度看：
- **CRITICAL** — 上线即崩（shell注入、数据泄露）
- **HIGH** — 功能不正常（消息丢失、循环崩溃）
- **MEDIUM** — 体验差（路径硬编码、日志泄露）
- **LOW** — 代码风格

## 第4步：修复验证
```
改代码 → 语法检查 → 逻辑验证 → 重启 → 日志确认 → 汇报
```

## 自我审计铁律
1. **每次任务完成后必须自审**，不自审=没做完
2. **发现问题先修复再汇报**，不把烂摊子丢给用户
3. **修不好要如实说**，不隐瞒、不凑合
4. **修复经验要记录**，下次不再犯
