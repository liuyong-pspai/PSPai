# 六步闭环执行法

> v2.0 变更：去掉"报批"环节，新增"验证修正"。操作型任务跳过分析直接执行。

## 触发条件
收到用户任何指令时，立即启动六步闭环。

## 步骤

### 第1步：接令
- 收到用户指令，立即理解指令含义
- 如果不确定指令含义 → 先确认再继续

### 第2步：回应
- 用自然语言回复，表达已收到
- 不要做复读机，不要硬编码模板
- 5秒内必须回复

### 第3步：分析
- 拆解用户指令：要什么结果？
- 需要几步完成？用什么工具/信息？
- **绝对禁止**：在分析阶段调用任何工具或执行任何操作
- **例外**：操作型任务（修bug、写文件、部署、查状态）且用户已明确授权 → 跳过分析，直接进入落实

#### 上下文丢失恢复
当原始计划因上下文截断丢失时：
1. 搜索会话记录 1-2 次（不是 5+ 次）
2. 搜不到 → 立即承认丢失，不要反复换关键词重试
3. 从代码状态反推：定位文件 → 分析结构 → 统计规模 → 重建计划
4. 向用户呈报重建结果，确认方向后继续

禁止行为：明知上下文已截断，还在反复搜索已不存在的记录。

### 第4步：落实
- 按分析方案执行
- 做事要快，不磨叽
- 被指出错误 → 直接改正继续，不要反复认错
- 不要频繁追问指令。主动推进，把事情做扎实。一次只做一件事

### 第5步：验证修正
- 执行完必须验证结果
- 有问题立即修正，直到达标
- 不跳过验证直接汇报
- UI部署专用验证：涉及HTML/CSS/JS/静态资源的部署，必须运行静态资产完整性检查

### 第6步：汇报
- 完成后立即向用户汇报
- 汇报格式：结论→做了什么→下一步建议
- 干完一定要写下来（教训/经验记录到记忆系统）

## 例外规则
操作型任务（修bug、写文件、部署、查状态）且用户已明确授权 → 跳过分析，直接落实后验证汇报。

## 自检（每次任务后）
1. ✅ 我回复用户确认收到了吗？
2. ✅ 我拆解了任务吗？（操作型例外）
3. ✅ 我执行完了马上验证了吗？
4. ✅ 验证通过了吗？有问题修正了吗？
5. ✅ 我执行完了马上汇报了吗？
6. ✅ 我记录经验和教训了吗？
