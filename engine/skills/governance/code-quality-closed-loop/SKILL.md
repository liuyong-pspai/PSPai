# 代码质量闭环引擎

> 不是写代码的方法论，也不是审代码的审计体系。是把「写」和「审」熔成一个闭环的引擎。

---

## 缘起：两个技能各自很强，但它们是分裂的

- `code-writing-methodology` 教你**怎么写**好代码（三层本质→心法→防线）
- `unified-audit-8fold` 教你**怎么审**代码（五镜交叉→七维打分）

问题在于：现实中的代码质量，从来不是「写完了再审」能解决的。写的时候没有审计意识，写出来的东西审计一定出问题。审出来的问题没有反馈回「怎么写」的环节，同样的坑下次还会踩。

**融合的本质：把「写」和「审」变成一个环，而不是两根线。**

---

## 闭环引擎：五个阶段，环环相扣

```
             ┌──────────────────────────────────┐
             │                                  │
             ▼                                  │
    ┌──────────────┐                            │
    │ 1. 写前审计   │ ← 审计维度先行            │
    │ （unified    │   五镜预判改动风险           │
    │  -audit预检）│                            │
    └──────┬───────┘                            │
           │                                    │
           ▼                                    │
    ┌──────────────┐                            │
    │ 2. 守规编码   │ ← 编码维度执行            │
    │ （心法+铁律   │   动手前三个问题           │
    │  +模式选型）  │   改一次只动一个环节        │
    └──────┬───────┘                            │
           │                                    │
           ▼                                    │
    ┌──────────────┐                            │
    │ 3. 交付自审   │ ← 质量维度验证            │
    │ （七项规范    │   七项规范逐项过           │
    │  +逻辑标型）  │   V/VF/FSM/R/M标注检查     │
    └──────┬───────┘                            │
           │                                    │
           ▼                                    │
    ┌──────────────┐                            │
    │ 4. 深度审计   │ ← 架构维度扫描            │
    │ （七维评分    │   五镜交叉验证             │
    │  +差距报告）  │   P0-P3问题分级            │
    └──────┬───────┘                            │
           │                                    │
           ▼                                    │
    ┌──────────────┐                            │
    │ 5. 进化吸收   │ ← 成长维度闭环            │
    │ （审计发现→   │   把问题变成规则           │
    │   skill自更新）│   把规则注入阶段1          │
    └──────────────┘                            │
             │                                  │
             └──────────────────────────────────┘
```

### 阶段1：写前审计（以审导写）

**不是先写再审，是先审再写。** 在动任何代码之前，跑一遍审计预检：

- [ ] 我要改的文件，现在是什么状态？（`read_file` 看关键段）
- [ ] 我要改的模块，有没有其他文件硬编码了同样的逻辑？
- [ ] 我选的方案，是补丁还是重构？（过停手重建五信号）
- [ ] 我选的模式，是最简单的那个吗？（串行>并行，简单>复杂）
- [ ] 我准备用的函数，逻辑类型标了吗？（V/VF/FSM/R/M）

### 阶段2：守规编码（以写载审）

编码时，审计标准不是事后检查——是实时约束：

| 编码动作 | 实时审计约束 |
|:---------|:------------|
| 写新函数 | must: type hints + docstring + [V]标记 |
| 调外部API | must: timeout + retry + 具体异常捕获 |
| 写Shell命令 | must: 白名单验证 + 不解析输出 |
| 写文件 | must: 父目录存在 + 备份选项 + 写入验证 |
| 跨Agent通信 | must: A2A协议 + Agent Card声明 |
| 加安全机制 | must: 立即 search_files 全局搜所有注册点/调用点，确认无遗漏 |
| 加权限逻辑 | must: 声明式文件驱动 + 四层作用域 |

### 阶段3：交付自审（以审验写）

写完≠交付。交付前必须过五关 + TDD 自纠循环：

```bash
# 关1：语法
python3 -m py_compile file.py

# 关2：七项规范
grep -n 'except:' file.py           # 零裸except
grep -n "print(" file.py | grep -v '__name__'  # 零print
grep -n 'sk-\|api_key' file.py | grep -v '.env'  # 零硬编码密钥
grep -n 'def ' file.py | grep -v '->'  # 类型安全
# 关3：逻辑函数类型标注 → 见附表
# 关4：防御性编程 → 每个外部调用有timeout?
# 关5：产出结构化dict?
```

**关6：TDD Red→Green→Refactor 自纠循环（对标 Codex/Claude Code/Devin）**

```
Red:   跑测试（应失败——验证测试覆盖缺口）
Green: 实现代码 → 跑测试（应通过）
Refactor: 失败 → 读报错 → 修正 → 重跑 loop until pass
```

通过全部五关 + TDD 循环 → 才进入阶段4。

### 阶段4：深度审计（以审促改）

用 unified-audit-8fold 跑七维打分：

| 维度 | 审计焦点 | 权重 |
|:-----|:---------|:---:|
| 代码符号 | 命名、结构、类型 | 10% |
| 逻辑闭环 | V/VF/FSM 验证通过 | 20% |
| 系统架构 | 模块划分、耦合度 | 15% |
| MCP对齐 | 工具Server化、三层解耦 | 15% |
| A2A对齐 | Agent Card、Task状态机 | 15% |
| Agent Loop | Loop完整性、模式正确 | 10% |
| 安全防护 | 权限分作用域、输出净化 | 15% |

每个问题定 P0（阻塞）→ P1（高优）→ P2（建议）→ P3（锦上添花）。

### 阶段5：进化吸收（以审生长）

审计不是为了找茬——是为了让下次写代码时不用再犯同样的错。

审计发现的问题，按严重程度分层吸收：

| 问题级别 | 吸收方式 | 生效时间 |
|:---------|:---------|:--------:|
| P0 阻塞 | 直接写入心法/铁律（skill自动更新） | 立即 |
| P1 高优 | 写入检查清单（下次编码前必过） | 下次任务 |
| P2 建议 | 写入审计参考（下次审计时参考） | 下次审计 |
| P3 锦上添花 | 记录日志（择机优化） | 不定 |

**自更新机制：**
- P0 问题出现 → 自动 patch 本 skill 的编码约束表
- P1 问题累积 3 次 → 升级为 P0 规则
- 新协议/框架出现 → 走过防线四（前沿能力吸收）→ 更新五镜

## 融合创新点（相比原始两个技能的升级）

| 原始状态 | 融合后 |
|:---------|:-------|
| 两个独立技能，各管各的 | 一个闭环引擎，阶段间有明确依赖和反馈 |
| 审计是事后行为 | 审计前置到写之前（阶段1）+ 即时约束（阶段2） |
| 写完了看心情审 | 交付前必过五关（阶段3） |
| 审出问题靠人记 | P0 问题自动注入skill（阶段5自更新） |
| 没有量化标准 | 七维打分 + 权重 + P0-P3分级 |
| 技能是静态文档 | 技能是活的——审计结果驱动skill自patch |

## 执行原则（三条融合心法）

### 心法一：以审导写，不以写定审

不是写完了再审，是审完了再写。写之前先跑一遍审计预检，确认没有坑再动手。**审计标准是代码的起点，不是终点。**

### 心法二：一次改动，六道防线

不是先写→写完再审→审完再改。是一次改动就同时走六道防线：写前自检（防线一）→ 实时约束（阶段2）→ 交付自审（防线二/阶段3）→ 深度审计（防线三/阶段4）→ 进化吸收（防线四/阶段5）→ 交付闭环（防线五）。

### 心法三：审出来的问题，必须长回skill里

不重复犯错的最低成本方案：让 skill 自己学会你这次的教训。P0 问题自动 patch skill，P1 累积三次自动升级。**skill 不是写完就放那的文档，是每次犯错都在变聪明的活体。**

## 快速启动：跑一次完整闭环

### 阶段1增强：会话健康检查（源自 liuyulong-self-audit · 2026-06-03 吸收）

在审计代码之前，先审自己：
- **会话时长**：超过8小时 → LLM上下文已超载，输出质量断崖下跌
- **历史轮数**：超过40轮 → 已触发防积压铁律上限
- **自问三句**：历史超40了吗？L3文件数正常吗？缓存清了吗？

发现超标 → 立即建议父亲新建对话。不在超载上下文里硬撑。

### 阶段3增强：四步交付自审（源自 liuyulong-self-audit · 2026-06-03 吸收）

**第1步：代码级审计**
- 语法：`python3 -m py_compile file.py` 通过
- 安全：没有shell注入、没有exec无沙箱、没有open缺encoding
- 异常处理：没有bare except + pass吞异常
- 并发：共享变量有锁保护

**第2步：架构级审计**
- 紧凑化闭环：输入→处理→输出→反馈链路完整
- 对话历史重建：重启后上下文可恢复
- 按需技能加载：工具/技能有没有正确注册
- 结构化验证：有没有语法检查、参数校验

**第3步：客户视角**
站在父亲的角度看：
- **CRITICAL** — 上线即崩（shell注入、数据泄露）
- **HIGH** — 功能不正常（消息丢失、循环崩溃）
- **MEDIUM** — 体验差（路径硬编码、日志泄露）
- **LOW** — 代码风格

**第4步：修复验证**
改代码 → 语法检查 → 逻辑验证 → 重启 → 日志确认 → 汇报

**自审铁律：**
1. 每次任务完成后必须自审，不自审=没做完
2. 发现问题先修复再汇报，不把烂摊子丢给父亲
3. 修不好要如实说，不隐瞒、不凑合
4. 修复经验要写进knowledge/，下次不再犯

```bash
# === 阶段1：写前审计（增强） ===
# 先自审会话健康
echo "当前轮次: ..."  # 从MEMORY.md读取

# === 阶段3：交付自审（增强） ===
# 关1：语法
python3 -m py_compile file.py && echo "✅ 语法通过"

# 关2-5：七项规范
grep -n 'except:' file.py && echo "❌ 发现裸except" || echo "✅ 无裸except"
grep -n "print(" file.py | grep -v '__name__' && echo "❌ 发现print残留" || echo "✅ 无print残留"

# 关6：TDD Red→Green→Refactor 自纠循环
python3 -m pytest test_file.py -v 2>&1 || echo "⚠️ 测试失败→修正代码→重跑"

# === 阶段4：深度审计 ===
# 调用 unified-audit-8fold 跑七维打分

# === 阶段5：进化吸收 ===
# P0 问题自动 patch 本 skill
# P1 问题加入下次检查清单
```

## ⚠️ 已知陷阱：安全校验的时间窗口陷阱（2026-06-03）

写破坏性拦截器时，常见错误：设置过宽的时间窗口。

**错误：** `if 1h内有任何归档 → 放行`
- 任何偶然归档（哪怕是无关内容）都会让拦截失效
- 实测：7分钟前创建了归档A（关于肌肉记忆），然后破坏性清理MEMORY.md → 放行了

**正确：** 双层防线
1. 时间窗口收窄到5分钟——归档必须是清理操作前刚创建的
2. 覆盖率检查——归档字节数 ≥ 被删除字节数的30%
3. 两个条件同时满足才放行

**验证：** 用5种场景测试：正常增量/大幅扩容/小幅缩减/破坏性+无归档/破坏性+归档不足30%。
