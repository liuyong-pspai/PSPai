# 批量代码修复工作流（2026-05-30 审计修复实战）

## 前提

审计发现 100+ 项时，核心挑战不是改什么而是怎么改：
- 并行 patch 同一文件 = 冲突
- 不分批 = 验证困难（一编译一堆错误不知道哪个 patch 导致的）
- 改完不验证 = 累积错误

## 工作流

### 阶段1：审计（可并行）
- 用 `delegate_task` 并行审计不同模块（run_agent / tools / gateway / agent）
- 每个子代理返回结构化报告：P0-P3 分级 + 具体行号 + 修复代码片段
- 合并报告，交叉验证 P0 发现（详见 unified-audit-8fold 措辞铁律）

### 阶段2：修复（必须串行）

按优先级分批，每批 3-5 项：

```
批次模板：
1. read_file 读准所有要改的位置（并行读取 OK）
2. patch × N（串行，每 patch 一个文件）
3. py_compile 验证所有本批修改文件
4. 全部通过 → 进下一批
5. 有失败 → 定位修复 → 重验 → 通过才进下一批
```

### 经典批次结构（55项修复的实际顺序）

| 批 | 类别 | 项 | 理由 |
|:--|------|:--|------|
| 1-2 | P0 安全+逻辑 | 11 | 安全第一，逻辑断链造成实际故障 |
| 3-4 | P0 数据+配置 | 7 | 数据完整性和配置一致性是基础 |
| 5-6 | P1 高优 | 6 | 注入防线/权限/日志泄露 |
| 7-8 | P2 竞态/TOCTOU | 8 | 低频但后果严重 |
| 9-10 | 注入防线补全 | 7 | 网关和提示构建是注入主入口 |
| 11-12 | 性能+完整性 | 7 | 线程池/SQL去重/安全默认值 |
| 13-14 | 收尾 | 4 | 重试逻辑/去重/日志 |
| 15 | 锦上添花 | 5 | import顺序/去硬编码/尾截断扫描 |

### 阶段3：全量验证
```bash
for f in 所有修改文件; do
    python3 -m py_compile "$f" || echo "❌ $f"
done
```

## 并行 vs 串行的选择

| 阶段 | 策略 | 原因 |
|:-----|:-----|:-----|
| 审计 | delegate_task 并行 | 不同模块独立，无共享状态 |
| 修复 | 串行 patch | 同一文件不能被两个 patch 同时修改 |
| 验证 | 串行 py_compile | 失败时需要精确归因 |

## 常见坑

1. **patch 转义问题**：old_string 中的 `\"` 会被工具误解析。解决：用 `read_file` 读原文后不带反斜杠传入。
2. **残留代码**：patch 的 old_string 匹配到旧循环但新代码可能留下重复逻辑。解决：改后 `read_file` 检查上下文。
3. **缩进漂移**：注释行缩进与原代码不一致导致语法错误。解决：严格对齐原文件缩进。
4. **大文件风险**：`run_agent.py` 9600+ 行，每条 patch 都要确认行号偏移。解决：每次 patch 后不依赖旧行号，重新 read_file 定位。

## 与统一审计的衔接

审计输出的每个 P0 发现必须带有：
- 具体行号（可定位）
- 修复代码片段（可直接 patch）
- 所属批次建议（P0安全优先于P0逻辑优先于P1）
