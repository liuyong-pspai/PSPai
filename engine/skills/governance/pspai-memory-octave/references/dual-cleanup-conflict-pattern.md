# 双重清理机制冲突模式

> 来源：L6悟道#004（2026-06-03）
> 性质：跨层发现的系统缺陷模式
> 关联：`⚠️ 对话清理破坏性陷阱` 章节（pspai-memory-octave SKILL.md）

---

## 模式定义

记忆系统存在**两种完全独立的清理机制**，走不同的代码路径，彼此不知对方的存在。当两者竞争时，更快的机制（对话清理）会抢先执行，跳过归档步骤，导致知识丢失。

```
机制A: L7 Cron清理     →  归档→指针→索引   （慢，守铁律）
机制B: 对话清理        →  全量替换           （快，违反铁律）
```

## 碰撞条件

两种机制在以下条件同时满足时发生冲突：
1. MEMORY.md 在对话中被大量写入（如产品决策、里程碑、架构讨论）
2. 上下文压力升高，agent 在 MEMORY.md 中写入「清理标记」请求下一轮精简
3. 下一轮对话启动时，上下文仍然紧张 → agent 执行 `write_file` 全量替换
4. L7 Cron 尚未运行（每周一次）或被跳过 → 没有机会在删除前归档

## 碰撞后果

| 损失类型 | 2026-06-02 实例 |
|:-----|:-----|
| 产品决策丢失 | 小龙人产品战略、Computer Use 引擎规划、PWA 发布路线 |
| 架构决策丢失 | 多语言架构 v2、发布双轨制（开源/闭源分离） |
| 质量规范丢失 | UI 交付标准、三界主题约束、爸的质量要求 |
| 环境凭证丢失 | Gitee 账号/密码/仓库路径 |
| 工程里程碑丢失 | 测试成绩（34 文件/77 passed）、v1.1.0 发布记录 |
| 修复记录丢失 | DGX-2 故障修复步骤 |

## 根因链

```
MEMORY.md 是「无保护的写入目标」
  → 任何 agent 可执行 write_file 全量替换
    → 无需经过归档检查
      → 对话清理（上下文压力驱动）比 L7 Cron（定时驱动）更快
        → 快机制抢先执行 → 慢机制失去归档机会
          → 知识丢失
```

## 安全护栏设计

### 必须实施（P0）

1. **写前检查**：任何 `write_file MEMORY.md` 如果新旧差异 > 50%
   → 必须先提取待删除条目 → 判断是否应进 L3 → 先归档再替换

2. **清理标记改写**：
   ```
   旧: "清理标记: 本对话上下文过长，需在下次会话时精简MEMORY"
   新: "迁移标记: 以下条目需迁移至L3后精简：{具体条目清单}"
   ```

3. **L7 审计增强**：每次 L7 扫描时
   → 比对 MEMORY.md 大小变化
   → 单次减少 > 50% 且 24h 内无对应新 L3 归档
   → 触发铁律违规告警 + 建议从会话历史补录

### 建议实施（P1）

4. **l1_memory_guard.py 增加 check_delete 模式**
   → `write_file` 前 `wc -c` 对比
   → 新旧差异 > 50% → 拒绝写入
   → 输出待归档条目清单

5. **L6 Cron 增加铁律违规检测**（已纳入 Cron 八步流程第 5 步）
   → 比对当前 MEMORY.md 与上次扫描时的大小
   → 差异 > 50% → 检查是否有对应新归档 → 无则告警

## 验证方法

```bash
# 检查 MEMORY.md 是否有异常大小波动
stat memories/MEMORY.md

# 检查是否有对应归档
ls -lt memories/archive/*.md | head -5

# 检查铁律违规告警
grep "铁律违规" memories/MEMORY.md
```
