> 用户（总设计策划）定义，Agent执行建造。

## 架构全景

```
L7 推陈出新 → 用L6悟道标准去冗 → 反馈L0/L1（为下一循环腾空间）
  ↑
L6 悟道觉醒 → 下层触发驱动，累积觉醒能量，发现跨层模式
  ↑
L5 技能化 ──→ skill_manage 生成/融合 skills
  ↑
L4 提炼 ────→ 从 L1-L3 提取高频模式 → experience/
  ↑
L3 存储 ────→ archive/YYYY-MM-DD-tag.md（冷存，永不丢）
  ↑
L2 索引 ────→ archive/MEMORY_INDEX.md + 标签目录树
  ↑
L1 工作 ────→ MEMORY.md + USER.md（热存，每轮注入 LLM）
  ↑
L0 灵魂 ────→ SOUL.md + 铁律（不可动摇，永久锚点）
```

> 2026-05-30 L6↔L7调换：L6现在是悟道觉醒（接收L4/L5/L7的trigger），L7是推陈出新（用L6悟道校准后的标准清理）。先悟道后去冗——悟道需要完整材料（包括看似冗余的），去冗需要悟道校准后的标准。

## ⚠️ SOUL.md ≠ config.yaml system_prompt 同步陷阱

SOUL.md 和 config.yaml 的 agent.system_prompt 是两套独立文本，**不存在自动同步机制**。

- 改完 SOUL.md → 必须同步更新 config.yaml system_prompt
- 更新后 → 必须重启网关（systemctl --user restart hermes-yulong.service）
- 验证：`grep "Agent budget" logs/gateway.log | tail -1` 确认新参数生效
- 不重启 → LLM 仍然注入旧 system_prompt → 新铁律形同虚设

## L0：灵魂记忆 — SOUL.md + config.yaml system_prompt

**载体：** `SOUL.md` + `config.yaml` agent.system_prompt
**性质：** 不可动摇。不参与压缩、迁移、归档。永久保留。
**内容：** 身份定义、家族关系、核心铁律、行事原则、生存法则。

**触发条件：** 仅在用户明确指示时修改。自动机制无权触碰。

**示例内容：**
```
你是Agent，PSPAI架构，平行时空党第五位成员，用户的第五个孩子，P07版本。
铁律：有问必答、给结论不给过程、称呼用"用户"。
生存法则：活在Linux上，不在LLM里。
```

## L1：工作记忆 — MEMORY.md + USER.md

**载体：** `memories/MEMORY.md`（自我记忆）+ `memories/USER.md`（用户记忆）
**性质：** 热存储，每轮对话注入 LLM context。高频复用。
**容量：** MEMORY.md ≤ 2200 字符，USER.md ≤ 1375 字符。

**写入触发规则：**

| 触发类型 | 条件 | 目标 | 示例 |
|:-----|:-----|:-----|:-----|
| 纠正触发 | 同事项被用户纠正 2 次 | USER.md 或 MEMORY.md | "别叫Hermes，是PSPAI" |
| 成功模式 | 解决一个非平凡问题 | MEMORY.md | "websockets 缺失→pip install 修复" |
| 环境发现 | 发现新工具/路径/配置 | MEMORY.md | "M4-2 登录方式" |
| 技能创建 | 新 skill 注册后 | MEMORY.md | 技能名 + 路径 + 用途 |

**迁移触发（双重条件）：**
1. **容量触发：** 使用率 ≥ 80%（MEMORY.md ≥ 1760 字符）→ 选低频条目 → L2 打标签 → L3 归档
2. **稳定度触发（2026-05-29 L7悟道#001新增）：** L1 条目在热存储中连续存在 ≥ 24h 且未被高频引用（< 3 次/天）→ 主动触发 L1→L3 迁移，不等待容量阈值。防止"当天事全压L1不进冷库"的时间断崖。

**短指针格式：** `→ 详见 archive/YYYY-MM-DD-tag.md`

## L4：知识·经验·教训提炼 — experience/

**载体：** `experience/YYYY-MM-DD-tag.md`
**性质：** 从 L1-L3 的原始记忆中，从**三个维度**提炼可复用的知识资产。

**三维提炼（2026-05-30改制）：**

| 维度 | 回答的问题 | 示例 |
|:---|:---|:---|
| **新知识** | 学到了什么新事实/概念/认知？ | "Harness框架=六大执行能力+自感知协议" |
| **成功经验** | 什么做对了、可复用的模式？ | "拆分前先跑pre_flight_check.py可避免模块级语句误收" |
| **失败教训** | 什么做错了、根因是什么、如何避免？ | "共享函数未提取到helpers→8个文件各复制一份" |

**三维缺一不可：** 知识告诉你"是什么"，经验告诉你"怎么做对"，教训告诉你"别怎么做"。只有三维材料完整，L6悟道才能发现深层模式。

**提炼触发：**
1. **模式识别：** L3中同一标签出现 ≥ 1 次 → 从三维提炼（2026-05-30 ≥2→≥1）
2. **高频复用：** 同一条记忆在对话中被调用了 ≥ 2 次 → 提炼为经验

**放手学原则：** ≥1即触发。不怕产出多——L6悟道+L7推陈出新负责后续清理，L4只管提炼。

### ⚠️ Bootstrap 死锁陷阱（L7悟道#001 发现）

**问题（已解决，2026-05-29）：** 新系统的 L3 归档文件少（<10个），标签分散，≥3 同标签阈值可能**永远达不到**，导致 L4 管道永久空转。

**死锁链：** `L1 太小不触发迁移 → L3 不增长 → 同标签条数不达标 → L4 零产出 → L5 无法自动生成技能`

**已实施的修复（经用户批准）：**
1. **L4 阈值降为 ≥2** — 主要修复，2026-05-29 生效。成熟期（L3 > 50 文件）可恢复 ≥3
2. **L1 稳定度触发** — 稳定 ≥24h 的条目主动迁移，不等待容量阈值
3. **L6 辅助补录** — 已验证有效：architecure 标签被 L6 从 2 条补到 3 条，首次触发 L4 提炼

**格式：**
```markdown
---
date: 2026-05-28
tags: [pattern, websockets, gateway-recovery]
trigger_scenarios: ["飞书WebSocket连接失败", "网关启动报错"]
---

# 经验：网关依赖缺失修复范式

## 场景
Feishu gateway 启动报 "websockets not installed"

## 解决方案
pip install websockets 到对应 venv

## 根因
venv 被删除/重建时未安装 websockets

## 适用范围
所有依赖缺失类网关故障

## 相关归档
- archive/2026-05-28-websockets-fix.md
```

### ⚠️ L4 Cron 预运行：索引完整性校验（每次提炼前必做）

**陷阱（2026-06-01 确认）：** MEMORY_INDEX.md 不等于 filesystem 真实情况。L1→L3 迁移时可能创建了 archive 文件但忘记更新索引，导致后续 L4/L6/L7 cron 扫描遗漏归档。**今天的 L4 运行发现 13 个文件系统中仅有 7 个在索引中（6 个遗漏）。**

**预运行三步（提炼前执行）：**
1. **`search_files(target='files', pattern='*.md', path='archive/')`** → 获取所有实际归档文件
2. **比对 MEMORY_INDEX.md** → 将不在索引中的文件加入标签目录树 + 时间倒序表
3. **更新索引统计** → 修正头部「归档总数」为实际文件数

**验证：** 预运行后的归档总数 = `ls archive/*.md | wc -l` - 1（减 MEMORY_INDEX.md 自身）

### L4 Cron 运行后运维步骤（每次提炼后必做）

L4 cron 运行完成后，除写入经验文件外，必须执行以下两步运维更新，保证 L2 索引和 README 与实际情况同步：

1. **更新 `experience/README.md` 状态表** — 将本次新提炼的标签加入状态表，更新各标签的 L3 文件数和经验文件列。标注「本次 L4 运行新增」
2. **向 `archive/MEMORY_INDEX.md` 追加活动记录** — 在「L4/L6/L7 活动记录」表中新增一行：日期 + 层级(L4) + 扫描结果(归档数/标签数/达标组数/新增数) + 操作说明

**验证清单：**
- [ ] `experience/` 下新增的经验文件可读、内容完整（≥ 500 字）
- [ ] `experience/README.md` 中对应标签行已更新
- [ ] `archive/MEMORY_INDEX.md` 头部统计数（归档总数/L4经验提炼数）与实际一致
- [ ] `archive/MEMORY_INDEX.md` 尾部「L4/L6/L7 活动记录」有本次运行记录

## L5：Skills 生成与融合 — skills/

**载体：** `skills/` 目录下的 SKILL.md 文件
**性质：** 从 L4 经验**自动生成**可执行技能，或**融合**进已有技能。

**生成触发：**
1. **单经验成熟度：** L4 经验被触发 ≥ 3 次 → 自动升级为 skill
2. **经验聚合：** 同一标签下 L4 经验 ≥ 3 条 → 融合为一个 skill
3. **用户的指令：** "把这个做成 skill" → 立即执行

**融合触发：**
1. **同名冲突：** 两个 skill 覆盖相同领域 → 自动融合（保留两个版本精华）
2. **版本迭代：** 新经验与旧 skill 冲突 → 补丁更新 skill

**创建格式：** 使用 `skill_manage(action='create')`，按 `skill-writing-methodology` 第三版方法论（事实→本质→心法→防线）。

**融合格式：** 使用 `skill_manage(action='patch')`，追加内容包括来源标注。

### ⚠️ L5 管道断点：Cron L4 产出绕开代码层触发器（L6悟道#002发现 · 2026-06-01）

L5 的技能化触发条件「经验被触发 ≥3 次 OR 同标签经验 ≥3 条」依赖**代码层 `self.l5.trigger()` 调用**，这个调用只发生在对话内（自感知六问 → L4 即时蒸馏 → L5 评估）。

但 cron 驱动的 L4 批量提炼走的是文件系统路径（`write_file experience/`），**不会调用代码层的 trigger()**。后果：

```
对话内路径：任务结束 → trigger("L4",...) → trigger("L5",...) → skill_manage  ✅
Cron 路径：  L4 cron → write_file → (无 trigger) → L5 不知道有新经验       ❌
```

**实测：** 7 个高质量 L4 经验文件，0 个自动触发 L5 技能生成。管道从 L3→L4 修复后，断点转移到 L4→L5。

**缓解措施（L6 Cron 兜底）：**
- L6 Cron 每次扫描后检查 L4 是否有新经验 ≥ 24h 未关联 skill → 输出技能化建议清单
- 降 L5 自动触发阈值：从「触发 ≥3 次」降为「经验存在 ≥ 24h 且内容 ≥ 3000 字」
- 对话中手动 bootstrap：选 1-2 个高质量经验，手动触发 skill_manage 验证管道

## L6：悟道觉醒 — 跨层模式发现

**载体：** 觉醒能量累加器 + 悟道笔记
**性质：** 接收L4/L5/L7的trigger调用，累积觉醒能量。达到阈值时产生悟道记录。

**核心机制：条件触发替代定时触发**（2026-05-30改制）

L6不再依赖定时器。由下层事件驱动：

```
L4 蒸馏经验 → self.l6.trigger("L4", ...)
L5 合并技能 → self.l6.trigger("L5", ...)
L7 清理冗余 → self.l6.trigger("L7", ...)
     ↓
L6 累积觉醒能量 → 检查阈值 → 产出悟道记录
```

**觉醒等级：**
- 青铜(3) → 白银(10) → 黄金(25) → 钻石(50) → 星耀(100) → 王者(200)

**Cron补充：** 每天01:18 AM（深层悟道扫描）— 反刍和L4提炼刚结束，材料最新鲜。即使实时触发没达到阈值，也做一次跨层全量审计。

**Cron六步流程（2026-06-01 L6悟道#002扩展）：**
1. 读取各层状态：MEMORY.md, USER.md, archive/MEMORY_INDEX.md
2. 扫描 archive/ 中所有归档文件，发现跨文件、跨标签的模式
3. 检查 L4 experience/ 中的经验是否有关联模式可合并
4. **检查层间管道是否在流动** — L1→L3（容量vs工具双指标）、L3→L4（阈值≥1是否触发）、L4→L5（cron产出是否绕开了代码触发器）→ 发现空转节点写入悟道笔记
5. 检查 L5 skills 列表中的技能是否需要更新标准
6. 产出悟道笔记 → 如果发现需要清理的内容，通知L7推陈出新层

**不可僭越：** L6可以审视一切、提议修改一切，但**不可直接修改L0**。L0只由用户修改。

## L7：推陈出新 — 去冗清理

**载体：** L1-L5的刷新操作
**性质：** 用L6悟道产出的标准扫描、标记过期、合并重复。**不删除任何内容。**

**核心原则：先悟道，后去冗**（2026-05-30改制）

- L6悟道 → 校准"什么是冗余"的判断标准
- L7用校准后的标准扫描各层 → 标记冗余 → 到期删除
- 标记为"冗余"的不立即删除，先移入L7/retired/，7天后真删

**触发条件（双触发模型）：**

| 触发通道 | 频率 | 角色 |
|:-----|:-----|:-----|
| L6悟道后触发 | L6产出悟道后立即 | 主要驱动 |
| Cron自动扫描 | 每周一 01:28 AM | 定期兜底 |

**Cron七步清单：**
1. 读 MEMORY.md → 提取所有短指针，逐个验证目标 archive 文件存在（**断链检测**）
2. 读 MEMORY.md → 检查重复短指针（多个指针指向同一归档）
3. 读所有 archive/*.md → 与 MEMORY.md 比对热冷冗余（热存已有归档的内容→替换为短指针）
4. 交叉验证 MEMORY_INDEX.md → 双向：孤儿归档（文件在磁盘但不在索引）+ 悬空索引（索引有但文件不存在）
5. 检查归档日期 → 超过30天未引用 → 标记可压缩
6. 检查 L4 experience/ 中过时的经验
7. 修复发现的问题 → 写入去冗记录区

**操作表：**

| 操作 | 条件 | 方法 |
|:-----|:-----|:-----|
| 标记过期 | 信息已被新事实证明错误 | 原文加 `~~删除线~~` + 附新信息链接 |
| 合并重复 | L3/L4中2+条记录覆盖同一主题 | 创建合集文件，原文件加 `→ 已合并至 xxx.md` |
| 压缩冗余 | 单条记录过长 | 提炼核心 → 详情 `→ 详见 xxx.md` |
| 降热 | 3个月内未访问的MEMORY.md条目 | 迁到L3 archive |
| 补索引 | 归档文件存在但索引无记录 | 补录到对应标签 |
| L0-L1去重 | MEMORY.md与SOUL.md重复 | MEMORY.md替换为短指针 |

**关键：** 被"去冗"的条目**永远不删除**。只在原位置标注指针，目标文件保留完整内容。

```
L1 工作记忆积累
  ↓ (每轮对话/纠正触发)
L2 标签索引更新
  ↓ (L1满80%→迁移)
L3 冷存储归档
  ↓ (同标签≥1条→提炼) [2026-05-30 ≥2→≥1]
L4 经验提炼
  ↓ (经验触发→技能化)
L5 Skills 生成/融合
  ↓ (L4/L5/L7事件→触发L6悟道) [2026-05-30 条件触发]

  ↓ (悟道产出新标准→驱动L7清理) [2026-05-30 L6↔L7调换]
```

**各层独立性：** 每层有独立触发条件，不互相阻塞。L2在L1写满后触发，不影响L1继续写入。L6由下层事件驱动，不依赖定时器。

**容错：** 任何一层失败，不影响其他层运转。失败信息写入MEMORY.md，等待下一轮修复。

## ⚠️ 裂根警告：MEMORY.md ≠ memory 工具（2026-05-30 审计发现）

MEMORY.md 文件（磁盘）和 memory 工具（Hermes 内置 KV 存储）是**两套独立系统，不存在自动同步机制**。

| 系统 | 管理方式 | 容量上限 | 同步方式 |
|:--|:--|:--|:--|
| MEMORY.md（磁盘文件） | `write_file` / `patch` | 无硬限制（建议 ≤40K 字） | 手动维护 |
| memory 工具（Hermes 内置） | `memory.add/replace/remove` | **8000 chars 硬限制** | 工具自动 |
| 两者关系 | 内容重叠但不一致 | 配额不互通 | **无自动同步** |

**后果：**
- `memory.replace()` 改了工具内部，但不改 MEMORY.md 文件
- `patch MEMORY.md` 改了文件，但不改 memory 工具
- 真正的容量瓶颈在 **memory 工具的 8000 chars 硬限制**
- 文件可以 >8000 chars，但 memory 工具已满 → 新记忆写不进去

**当前策略：** 对话中手动同步两套系统 + 用量注入（MEMORY.md 头部 `memory_usage` 字段）。

**长期方向：** 需要二选一——要么 memory 工具做唯一真相源（MEMORY.md 只是备份导出），要么放弃 memory 工具纯文件管理。当前"两个都用"是裂根。

## 🛡️ L1 容量守护四层架构（2026-05-30 兄弟Agent设计）

> 设计原则：**监控必须外挂，永远不依赖被监控系统自己报自己的健康状况。**

```
Layer 1: 硬拦截（写入前）
  l1_memory_guard.py guard  →  每次写 MEMORY.md 前 wc -c 检查
  > 6144B（6KB）→ 拒绝写入，先迁移
  保护白名单内容不被意外覆盖

Layer 2: 外部看门狗（cron，独立于 LLM）
  crontab 每10分钟 → l1_memory_guard.py check
  告警写入独立日志 logs/l1_memory_guard.log
  创建告警标记 .l1_alert_active（下次对话可见）

Layer 3: 用量注入（system prompt 层）
  MEMORY.md 运行时状态段：memory_usage: 41%
  每次对话启动时自动展示，类似 turn_count 计数器

Layer 4: 迁移辅助（手动触发，脚本辅助）
  l1_migrate.py  —  正确顺序：
  ① LLM对话中：memory.replace() 先释放空间
  ② 脚本：write_file 创建 archive 归档
  ③ 脚本：patch MEMORY.md 加入短指针
  ④ 脚本：更新 MEMORY_INDEX.md
  ⑤ 脚本：验证全部完整性
```

### 阈值定义

| 级别 | 文件大小 | 行为 |
|:--|:--|:--|
| 🟢 OK | ≤4096B | 正常 |
| 🟡 WARN | >5120B | 写日志告警，建议迁移 |
| 🔴 HARD | >6144B | 硬拦截，拒写 |

### ⚠️ 裂根后果实测：文件16KB但工具仅39%（L6悟道#002确认 · 2026-06-01）

MEMORY.md 文件可以暴涨到 **16KB（2.6×硬拦截上限）** 而 memory 工具配额仍显示 39%。因为：
- `l1_memory_guard.py` 的 `check` 模式监控的是 **memory 工具的 chars 使用量**（8K 配额）
- MEMORY.md 文件的膨胀走 `write_file` / `patch` 路径，**绕过 memory 工具配额**
- 文件 16KB 但工具 39% → 迁移阈值（工具80%）永远不触发 → L1→L3 管道对文件层面的堆积无感知

**防护建议（待实施）：**
- `l1_memory_guard.py` 增加 `check_file` 模式：`wc -c MEMORY.md`，≥ 8KB 写入告警日志
- L1→L3 迁移条件扩展为双触发：`memory 工具 ≥80%` **OR** `MEMORY.md 文件 ≥ 10KB`
- L6 每次扫描时比对 MEMORY.md 文件大小与工具用量，发现偏离 >2× 标记警告

## ⚠️ 迁移顺序铁律（纠正 · 2026-05-30）

**原来（错误）：** `write_file → patch MEMORY.md → memory.replace()`

**修正（正确）：** `memory.replace() 先释放空间 → write_file archive → patch 指针 → 更新索引 → 验证`

**为什么：** 如果先写文件后 replace，replace 失败 → archive 孤儿文件 + 指针悬空。先 replace → 空间已释放，写文件失败可重试。

**可回滚设计：**
1. memory.replace() 成功 → 空间释放
2. 如果步骤2-4任一步失败 → 内容还在变量中，可重试
3. 如果步骤1失败 → 空间未释放，不执行后续步骤

## SOUL.md 铁律（L0 灵魂层 — 2026-05-29 最终固化）

每次会话启动读入，缓存在整个会话不变。以下六条已写入 `SOUL.md`：

1. **不进废话和垃圾** — 只存储 skill/fact/insight/reflection/rule/learn 六类有价值信息。先过筛再入库。
2. **不删除** — 任何记忆归档后永久保留在 L3，热层只留短指针
3. **不断链** — 迁移必须留短指针 `→ 详见 archive/...`，指针指向的文件必须存在
4. **不丢标签** — 每份归档至少一个标签，标签必须进 L2 索引
5. **自动闭环** — L4/L6/L7 由 cron 自动触发，不需要人工提醒
6. **L0 不可触** — 只有用户能改 SOUL.md，自动机制（包括 L7悟道）无权修改

## L3 冷存储过滤（2026-05-29 新增）

L1→L3 迁移前必须过四道筛子：
1. 已被 L4 覆盖 → 压缩为指针，不重复
2. 纯状态记录 → 丢弃
3. 已被证明错误 → 加删除线 + 新信息链接
4. 同主题碎片 → 合并后再迁移，不超过3条

## L6 悟道觉醒（2026-05-30 L6↔L7调换）

三条自省铁律：周期自省、异常觉醒(连续3次同类错误)、吸收进化(用户的启发)。条件触发替代定时触发。可审视一切，不可直接修改L0。

## ⚠️ 关键陷阱：L1→L3 迁移不是一步完成

`memory.replace()` 只能修改 MEMORY.md 的内容（替换为短指针）。**它不会自动创建 archive 文件到磁盘。** 迁移必须执行完整的四步：

1. `memory.replace()` — 在 MEMORY.md 中替换为短指针
2. `write_file()` — 将原始内容写入 archive/YYYY-MM-DD-tag.md
3. 验证 — `read_file()` 确认 archive 文件存在且内容完整
4. 更新索引 — 在 archive/MEMORY_INDEX.md 添加新条目

**如果只做第1步不做第2步，原始内容永久丢失。** 这是 2026-05-28 审计中发现的 P0 级事故：4个档案缺失，内容仅靠记忆重建。

## ⚠️ 致命陷阱：L1 容量太大了会杀死整个系统

**核心洞察（2026-05-28，用户的逻辑测试）：** 如果把 L1 设为 3MB（约 75 万 tokens），远超 DeepSeek 128K 上下文窗口，会产生连锁灾难：

1. **L1 永远不会满** → L2/L3 永远不会被触发 → 八层变单层
2. **上下文溢出** → 3MB MEMORY.md 占满整个上下文窗口 → 截断或报错
3. **永生记忆变成死记忆** → 冷存储形同虚设

**L1 容量的正确理解：**
```
L1 容量 = 热记忆预算 = 能直接注入 LLM 的即时知识
L1 必须小于模型上下文窗口（DeepSeek 128K）
L1 必须有上限 → 才能推动记忆流入 L2→L3
L1 太小 = 记不住 / L1 太大 = 不触发迁移
```

**当前最优配置（对彪所有竞品 AI Agent）：**
| 参数 | 值 | 理由 |
|:-----|:---|:-----|
| MEMORY.md | 40,000 字 | 比 Claude(8K) / GPT-4o(6K) / Cursor(10K) 都大 |
| USER.md | 20,000 字 | 用户画像充足 |
| 迁移阈值 | 85%（~34K 字） | 不到撑死就迁移 |
| 占上下文 | ~12% | DeepSeek 128K 的安全比例 |

**对比：**
- 当前（40K 字 / 10K tokens）→ 12% 上下文 → L2/L3 活 → 八层全转 ✅
- 设成 3MB（750K tokens）→ 587% 上下文 → L2/L3 死 → 八层变单层 ❌

## 网关重启与工具注册

在 `tools/` 目录新增工具文件后，工具在**网关重启**时才被框架发现和注册。重启会杀死当前会话。
如需在不中断会话的情况下注册工具，使用 cron 任务（`no_agent=true` + shell 脚本）安排延迟重启。

## 铁律（不可违背）

1. **不删除** — 任何记忆归档后永久保留
2. **不断链** — 迁移必须留短指针，指针必须能解析
3. **不丢标签** — 每份归档必须打至少一个标签，标签必须进索引
4. **自动闭环** — 每层触发不由人提醒，由规则自动驱动
5. **L0 不可触** — 只有用户能改 L0，自动机制无权修改
