# 深度验证方法论：三层穿透式审计

> 来源：2026-06-03 爸两次要求"深入验证""别表面上能够通，背地里又是短路"
> 核心理念：表面能跑≠没问题。必须穿透代码→配置→运行时三层。

## 三层穿透模型

```
层1: 语法/结构层（表层）
  py_compile 通过、文件存在、格式正确
  → 最容易被验证，也最容易产生虚假安全感

层2: 逻辑/数据流层（中层）
  变量读取路径、配置链完整性、API调用序列
  → 跨文件追踪数据流，找断裂点

层3: 运行时/异常路径层（深层）
  skip_memory=True、模型硬编码、错误无重试
  → 模拟异常场景，找静默失败
```

## 验证铁律

1. **永远不只验证表层。** py_compile通过≠没问题。skip_memory=True能通过编译但杀了整个记忆系统。
2. **追踪每一个默认值到源头。** `DEFAULT_PROMPT = 'You are PSPAI.'` 这一行决定AI身份，必须追溯到它是从哪来的、有没有被覆盖。
3. **模拟异常路径。** 正常流程能跑=50%验证。引擎挂了前端怎么办？配置缺失引擎怎么降级？fetch失败有没有重试？
4. **双向同步必须逐条对比。** SOUL.md ↔ config.yaml 不能靠"印象中一致"，必须逐条关键词匹配。
5. **所有`==`和`!=`条件表达式的默认值必须逐行模拟。** 这是审计中最高频的误判来源。
6. **修复后立即回测破坏场景。** check_delete修复后必须测试"破坏性缩减-90%无归档"是否真的拦截。

## 实战案例

### 案例1：skip_memory=True 穿透
- 表层：py_compile通过，引擎能启动 ✅
- 中层：配置链正常，API key读到 ✅  
- 深层：skip_memory=True → 每次对话从零开始 → P0致命 🔴
- 发现方式：逐行读pspai_server.py，追踪每一个AIAgent()参数

### 案例2：DEFAULT_PROMPT溯源
- 表层：引擎有system_prompt ✅
- 中层：从config.yaml读取 ✅
- 深层：config.yaml不存在时→DEFAULT_PROMPT='You are PSPAI.'→无身份 🔴
- 发现方式：追踪 `_config_loaded=False` 分支

### 案例3：前端零容错
- 表层：HTML有665行，功能完整 ✅
- 中层：有fetch调用，有.catch ✅  
- 深层：0次error/retry/loading/disconnect/reconnect → 🔴
- 发现方式：全文搜索前端错误处理关键词，发现全零
